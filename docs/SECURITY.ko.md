### 포함할 내용

- **설명(Description)**: 취약점에 대한 명확한 설명
- **재현 단계(Steps to Reproduce)**: 문제를 재현하는 상세한 단계
- **영향(Impact)**: 취약점의 잠재적 영향
- **영향받는 구성 요소(Affected Components)**: 영향받는 크레이트 또는 모듈
- **환경(Environment)**:
  - 운영체제 및 버전
  - oxrdp 버전 / 커밋
  - 디스플레이 서버(X11 / Wayland) 및 데스크톱 환경
  - GPU / VA-API 드라이버(디코드 경로 문제의 경우)
  - 반대편의 RDP 서버 / Windows 버전

## 위협 모델

oxrdp는 RDP **클라이언트**입니다. 클라이언트 관점에서 연결 대상 서버는 **신뢰할 수 없는
네트워크 입력**입니다. 연결 시퀀스 PDU, 기능 집합(capability sets), 가상 채널 데이터,
GFX/RemoteFX/비트맵 코덱 비트스트림, RAIL 윈도 메타데이터, 클립보드 및 드라이브 리디렉션
페이로드 등 — 와이어를 건너오는 모든 바이트는 oxrdp가 파싱하며, 서버가 "당신 자신의"
Windows 게스트일 때조차도 잠재적으로 적대적인 것으로 취급되어야 합니다(게스트는 일반적인
Windows 측 활동을 통해 침해될 수 있습니다).

메모리 안전성이 핵심 방어선입니다: 프로토콜 스택, RAIL 상태 머신, 코덱 프레이밍은 안전한
Rust로 작성되어 있어, 잘못된 형식의 PDU는 메모리 손상이 아니라 타입이 지정된 파싱 오류를
산출합니다.

## 적용 범위

보안 신고의 적용 범위:

- 서버 입력에서 도달 가능한 PDU / 채널 / RAIL 파싱의 **메모리 안전성 결함**
  (복구 가능한 오류여야 하는 패닉, UB로 유도될 수 있는 모든 `unsafe`).
- **코덱 디코드 안전성** — 적대적 비트스트림이 주어졌을 때 H.264 / RemoteFX / 비트맵 디코드
  경로에서의 범위 초과 또는 UB(`openh264` / VA-API로의 FFI 경계 포함).
- **TLS / 인증서 검증 우회** — 구성된 신뢰 정책(TOFU / 핀 고정 / 시스템 CA)이 거부해야 하는
  서버 신원을 수락하는 것.
- **자격 증명 노출** — 비밀번호나 토큰이 로그, 디스크, argv, 또는 환경 변수에 유출되는
  방식으로 기록되는 것.
- `oxrdp-cli` 또는 winpodx IPC 제어 채널에서의 **명령/인자 인젝션**
  (서버 또는 게스트에서 유래한 문자열이 셸이나 argv에 도달하는 것).
- 드라이브 리디렉션(`\\tsclient`)에서의 **경로 탐색(Path traversal)** — 서버가 공유 루트를
  벗어나 임의의 호스트 경로를 읽거나 쓰는 것.

## 적용 범위 외

- 머신에 대한 물리적 접근이 필요한 공격.
- 사회공학(social engineering) 공격.
- 서드파티 의존성의 취약점(이는 업스트림 프로젝트에 신고하십시오;
  업스트림이 수정 사항을 배포하면 핀을 올릴 것입니다).
- 의도적으로 잘못 동작하도록 당신이 제어하는 서버로부터의 서비스 거부(자원 한도는
  winpodx 단일 테넌트 모델에서 신뢰 경계가 아니라 강화 수단입니다).

## 보안 모범 사례

- **안전한 Rust 코어**: 검토된 `// SAFETY:` 정당화 없이는 프로토콜 / RAIL / 채널 로직에
  `unsafe`가 없으며, 코덱 라이브러리로의 FFI는 격리되고 범위 검사가 이루어집니다.
- **신뢰할 수 없는 서버 자세**: 모든 서버 입력은 할당하거나, 디스크에 쓰거나, 명령줄에
  삽입되기 전에 크기가 제한되고 검증됩니다.
- **코드나 git에 비밀 없음**: 자격 증명, 토큰, 키는 절대 커밋되지 않습니다.
- **명시적 인증서 신뢰**: TLS 인증서 처리는 명시적이며(TOFU / 핀 고정 / 시스템 신뢰),
  기본 빌드에서 절대 조용히 모두 수락하지 않습니다.
- **argv 전용 서브프로세스**: 서버 또는 게스트에서 유래한 문자열을 `shell=true` 방식으로
  호출하지 않습니다.
